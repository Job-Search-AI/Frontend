# 1. 문제 요약
- 검색 요청이 동기(`/api/query`)로 처리되면서, 백엔드 처리 시간이 90~120초로 길어질 때 중간 게이트웨이(Netlify 경유)에서 `504/502`가 먼저 발생했다.
- 결과적으로 사용자는 정상 검색이 아닌 실패 화면을 반복적으로 보게 되었고, 재시도 시에도 동일 경로에서 실패가 누적됐다.

# 2. 영향 범위(서비스/사용자/기능)
- 서비스: 메인 검색 기능의 신뢰도 저하.
- 사용자: 장문/복합 질의에서 결과를 받기 전에 타임아웃 오류 경험.
- 기능: `검색 시작 -> 결과 렌더링` 핵심 플로우가 네트워크/게이트웨이 제한에 종속.

# 3. 재현 조건
- 프론트에서 `POST /api/query` 호출 후 결과를 한 번에 기다리는 동기 구조.
- 백엔드 실제 처리시간이 30초 이상 늘어나는 구간(혼잡/긴 처리).
- Netlify 경유 연결에서 중간 타임아웃이 먼저 발생하는 경우.

# 4. 가설 목록
- 가설 A: 프론트 자체 타임아웃(12초)이 1차 실패 원인이다.
- 가설 B: 프론트 타임아웃 제거 후에도 게이트웨이(플랫폼) 제한으로 실패가 남는다.
- 가설 C: 구조를 `접수+조회` 비동기로 전환해야 장기 작업을 안정적으로 수용할 수 있다.

# 6. 실패 원인 분석
- 핵심 실패 원인은 단일 동기 연결로 장기 작업 결과를 한 번에 기다린 구조였다.
- 백엔드 처리시간이 길어질수록 프론트/게이트웨이의 연결 한계가 먼저 도달했다.
- 즉, 타임아웃 숫자 조정만으로는 구조적 병목을 제거할 수 없었다.

# 7. 의사결정 근거
- 검색은 장기 처리 가능성이 높은 작업이므로, 요청-응답 1회 모델보다 작업 큐 기반 상태 조회 모델이 적합하다.
- 사용자 체감과 안정성을 동시에 위해 즉시 접수(빠른 응답) + 주기 조회(완료 시 렌더링) 방식 채택.
- 백엔드 전환 일정과 독립적으로 프론트는 비동기 API 계약을 먼저 수용하고, 프록시 실패 시 폴백 경로를 유지한다.
